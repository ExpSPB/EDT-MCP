# 1.41 — Manual Smoke Test Scenarios

Use-case-driven scenarios for the four 1.41 phases. Run each in EDT against
a real project. Mark each PASS / FAIL / N/A on the release-readiness
checklist.

For every scenario:
1. **Setup** — project state needed.
2. **Action** — MCP call to make.
3. **Expected** — what the response should look like.
4. **Verification** — how to confirm the change really happened.

> Open the EDT MCP server log (Eclipse → View → Console) before running.
> Failures and probe-not-found events surface at WARNING level.

---

## Phase 1: `find_references` Pending bounded executor

### 1.1 Pending JSON returned within timeout window

**Setup**
- Open any large 1C configuration where `Catalog.Сотрудники` (or any
  catalog with 30+ kLOC of BSL referencing it) is reachable.

**Action**
```
find_references
  projectName=YourProject
  objectFqn=Справочник.Сотрудники
  timeoutSeconds=10
```

**Expected**
- Response is a JSON Pending object with `status="Pending"`, a stable
  `runKey`, `elapsedMs ≈ 10000`, `waitedMs=10000`.

**Verification**
- Inspect the registry counter via repeated calls (same params should
  produce the same `runKey`).
- Repeat the same call with `runKey=<runKey>` — receive either the
  finished result or another Pending JSON. Repeat until done.

### 1.2 timeoutSeconds clamps

**Action 1**: `find_references projectName=X objectFqn=Y timeoutSeconds=999999`
**Expected**: `waitedMs=120000` (clamped to 120 s upper bound).

**Action 2**: `find_references projectName=X objectFqn=Y timeoutSeconds=1`
**Expected**: `waitedMs=5000` (clamped to 5 s lower bound).

### 1.3 Bounded executor backpressure

**Action**: launch 30+ parallel `find_references` calls (e.g. via a
script with concurrent threads).

**Expected**
- The MCP server does not OOM. Excess tasks queue up to size 20 and then
  invoke `CallerRunsPolicy` (the calling MCP HTTP-handler thread runs the
  task).
- No new background threads beyond `corePoolSize=2 + maxPoolSize=8 = 10`
  named `find-references-async-N`.

## Phase 2: `export_object` kind detection + Pending

### 2.1 ExternalDataProcessor → .epf

**Setup**
- Workspace contains an EDT project of type "External data processor"
  (V8ExternalDataProcessorNature).

**Action**
```
export_object
  projectName=MyExternalProcessor
  outputPath=C:/temp/MyTool.epf
```

**Expected** (when EDT export API is reachable)
- Response status "success", `detectedKind="ExternalDataProcessor"`,
  `discoveredApi=<service name>`, `elapsedMs<30000`.

**Verification**
- Open `C:/temp/MyTool.epf` in 1C:Enterprise — the tool launches without
  errors.

### 2.2 ExternalReport → .erf

Same as 2.1 with project nature `V8ExternalReportNature` and outputPath
ending in `.erf`. Verify the file opens in 1C:Enterprise.

### 2.3 Kind/extension mismatch

**Action**
```
export_object
  projectName=MyExternalProcessor   # ExternalDataProcessor nature
  outputPath=C:/temp/wrong.erf      # .erf extension
```

**Expected**
- Error response with `kindMismatch` tag carrying `requestedExtension`,
  `detectedFromNature`, `expectedExtension`.

### 2.4 Unsupported extension

**Action**: `outputPath=C:/temp/x.txt`

**Expected**: Error response naming `.txt` as unsupported, advising to
omit the extension or pass `.epf` / `.erf`.

### 2.5 EDT API not exposed

**Action** (any export call when none of the 7 candidates is loaded)

**Expected**
- Error response with `exportApiNotFound` tag, `phase="probe"`, and a
  `triedServices` array listing all 7 fully-qualified service names.

## Phase 3: Forms ops

### 3.1 addFormAttributeColumn

**Setup**
- A form with an existing `FormAttribute` named `Items` of type Table.

**Action**
```
edit_metadata operation=addFormAttributeColumn
  projectName=MyConfig
  formFqn=Catalog.Products.Forms.ItemForm.Form
  parentAttributeName=Items
  name=Description
  title=Описание
  dataPath=Items.Description
```

**Expected**
- Success response, `message="added column 'Description' to FormAttribute 'Items'"`.

**Verification**
- Open the form in EDT WYSIWYG editor — the new column should be visible
  inside the table-attribute.
- Repeat the same call → response contains
  `"addFormAttributeColumn idempotent: column 'Description' already exists in 'Items'"`.

### 3.2 addDynamicListTable

**Setup**
- An empty managed form (any existing empty form on a Catalog / Document).

**Action**
```
edit_metadata operation=addDynamicListTable
  projectName=MyConfig
  formFqn=DataProcessor.Tools.Forms.MainForm.Form
  attributeName=ItemList
  tableName=ItemListTable
  mainTable=Catalog.Products
  title=Список товаров
```

**Expected**
- Success with `attributeName=ItemList`, `tableName=ItemListTable`.

**Verification**
- Open the form in EDT. The new attribute appears with type DynamicList,
  `MainTable=Catalog.Products`. The new table is bound via `dataPath=ItemList`.

### 3.3 setupSettingsComposerOnForm

**Setup**
- A managed form on any Report.

**Action**
```
edit_metadata operation=setupSettingsComposerOnForm
  projectName=MyConfig
  formFqn=Report.SalesAnalysis.Forms.MainForm.Form
```

**Expected**
- Success response carrying:
  - `composerName="Composer"`
  - Two non-null `bslSnippetRu` and `bslSnippetEn` fields with the
    Composer.Initialize / .ЗагрузитьНастройки snippet referencing
    Schema (СхемаКД).

**Verification**
- Open the form in EDT — `Composer` appears as a FormAttribute of type
  DataCompositionSettingsComposer; two UI tables (`SettingsTable` and
  `UserSettingsTable`) are present.
- Paste the BSL snippet into `ProcedureOnCreateAtServer` and run the
  report — the composer initialises without errors.

### 3.4 formApiNotFound graceful fallback

If EDT 2026.1 does not expose any of the probed factories (rare), the
response carries `formApiNotFound` tag with `missingFactory` field
naming the absent method. The form is left untouched.

## Phase 4: DCS 13 deferred ops

### 4.1 Scenario A — clean schema with all 13 ops

**Setup**
- Create a Report with one DCS schema containing one DataSetQuery
  (`Catalog.Products` source, 3 fields: Name, Price, Quantity).

**Actions** (run sequentially)

```
edit_metadata operation=addSettingsTable name=MainTable
  projectName=...; objectName=Report.Sales
edit_metadata operation=addSettingsChart name=Pie
edit_metadata operation=addSettingsVariant name=ByMonth presentation=By month
edit_metadata operation=addSettingsFilterGroup groupType=AndGroup
edit_metadata operation=addSettingsOrder field=Price orderType=Desc
edit_metadata operation=addSettingsSelectedField field=Name title=Product
edit_metadata operation=addSettingsSelectedField field=Price
edit_metadata operation=setSettingsParameter name=DateFrom value=2026-01-01
edit_metadata operation=addUserField name=Discount expression="Price * 0.1"
edit_metadata operation=removeConditionalAppearance index=0 target=schema
edit_metadata operation=setDataSetFieldAppearance dataSet=DataSet1 field=Price textColor=Red
edit_metadata operation=setOutputParameter name=Title value="Sales report"
edit_metadata operation=repairReportSchema
```

**Expected**
- Each call returns success.
- The final `repairReportSchema` reports a clean schema.

**Verification**
- Open the report in EDT GUI — settings tree shows: 1 table, 1 chart,
  1 variant "ByMonth", 1 filter group AND, 1 order, 2 selected fields,
  1 user field "Discount". Output parameter "Title" is populated.

### 4.2 Scenario B — dryRun

For 5 of the ops above, repeat with `dryRun=true`.

**Expected**
- Each response status "success", message says "Dry run completed
  without applying changes".
- BM state unchanged (verify by reopening the report).

### 4.3 Scenario C — idempotency for variants

**Action**: `addSettingsVariant name=ByMonth ...` (twice)

**Expected**: second call returns `alreadyExists` tag with
`name=ByMonth, kind=settingsVariant`.

### 4.4 Scenario D — cascade remove

**Action**:
```
edit_metadata operation=removeSettingsItem itemPath=Structure[0]
```

**Expected**
- `message="settings item at 'Structure[0]' removed (cascade)"`. The
  table and any nested fields/groups under it are gone.

### 4.5 Scenario E — factory miss

**Setup**: an EDT runtime where `DcsFactory.createSchemaVariant` is
absent (synthetic test by pointing to a stripped-down EDT).

**Action**: `addSettingsVariant name=X`

**Expected**: Error response with `dcsFactoryMethodNotFound` tag,
`triedMethods="createSchemaVariant, createSettingsVariant"`.

## Build & Distribution

After all four phases pass, run a clean build:

```
/tmp/apache-maven-3.9.9/bin/mvn -f mcp/pom.xml clean verify -DskipTests
```

The deliverable ZIP appears at:
```
mcp/repositories/com.ditrix.edt.mcp.server.repository/target/
  com.ditrix.edt.mcp.server.repository-1.41.0-SNAPSHOT.zip
```

Install via Eclipse → Help → Install New Software → Add → Local Archive.
After EDT restart, the MCP server registers `66` tools (no count change
from 1.40, but Forms 3 + DCS 13 ops are now native instead of deferred).
